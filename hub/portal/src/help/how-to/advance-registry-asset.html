How to Advance a Registry Asset

From draft registration to champion in production.

Step 1 — Create the asset

Go to Registry and click Create Asset. Choose a kind — task (a single-step function) or agent (an orchestrating entity). Give it a name and an initial version number (e.g. 1.0.0). The asset is created in draft lifecycle stage.

Step 2 — Link to an approved intake

Before the asset can advance beyond staging, it must be linked to an intake that is both approved and obligation-resolved. Use the Link Intake action on the asset card. This is the governance gate — it ensures every production asset is backed by a reviewed and approved use case.

You can link the asset before the intake is approved; the gate is enforced at promotion time, not at linking time.

Step 3 — Staging

Advance the asset to staging for non-production testing. At this stage the asset can be deployed in test or pre-production environments. The linked intake must exist, but does not yet need to be resolved (this allows early integration testing before compliance sign-off is complete).

Step 4 — Challenger

Advance to challenger to run alongside the current champion in production in shadow mode (outputs logged but suppressed — no business effect) or A/B mode (outputs applied to a scoped sample). This requires the linked intake to be both approved and obligation-resolved. The governance gate enforces this at the point of promotion.

Step 5 — Champion

Once the challenger has been validated, advance to champion. The champion is the active production asset. Advancing a new challenger to champion automatically deprecates the previous one, which is retained in a locked, audit-only state.

Rollback

A deprecated champion can be restored via rollback if the new champion has a defect. The rollback is recorded in the asset's audit trail.
