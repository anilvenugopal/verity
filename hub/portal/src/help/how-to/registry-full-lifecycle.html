Full Registry Lifecycle

From first draft to champion — the complete promotion path for an agent or task.

Stage 1 — Draft

Every new version starts in draft. Add prompt assignments, tools, and bindings. No governance gate applies at this stage.

Stage 2 — Candidate → Staging

Advance through candidate and staging via the lifecycle endpoint or the Advance button. No gate check yet — these are internal preparation stages.

Stage 3 — Challenger (gated)

Advancing to challenger requires the executable to be linked to an intake that is approved with all obligations resolved. The promotion gate checks this automatically and returns a 409 with the blocking reason if the condition is not met.

Stage 4 — Champion (atomic swap)

Promoting to champion via POST /api/registry/versions/{id}/promote (or the Promote button on the version detail page) is an atomic operation: the previous champion is revoked in the same transaction, and the new version becomes champion. The entity_champion_current view always reflects the latest state.

At least one prompt assignment is required before promotion. The same gate (approved intake + resolved obligations) applies when advancing via the lifecycle endpoint with to_stage: champion.

Stage 5 — Deprecated

A champion can be deprecated once a successor is promoted. Deprecated versions remain in history for audit purposes.
